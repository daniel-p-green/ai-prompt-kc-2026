## Goal

Deliver a pilot-ready MVP that helps residents find usable food resources and helps coordinators target outreach in the highest-need Kansas City ZIP codes.

## Inputs

- `docs/discovery/idea-brief.md`
- `docs/product/prd.md`
- `docs/product/jtbd.md`
- `docs/product/use-cases.md`
- `docs/engineering/test-strategy.md`

## Success Contract

- Done means: a judge can demo one coordinator flow and one resident flow end-to-end using challenge data and see clear pilot metrics.
- Tested by: unit tests for ranking logic, integration tests for sync plus alert publish flows, and one E2E pass for the resident happy path.
- No changes to: full delivery logistics, partner ERP replacement, or broad case-management workflows.

## Execution Plan (3 Steps)

1. Scaffold the app, data model, and sync layer around the challenge API, then lock the ranking rules with tests.
2. Build the two essential product surfaces: operator dashboard and bilingual resident search plus alerts.
3. Instrument outcomes, run a seeded demo, and polish the submission narrative around pilot proof and city ROI.

## Task Breakdown

### Task 1

- Files to modify: `package.json`, `app/`, `lib/challenge-api.ts`, `lib/scoring.ts`, `lib/types.ts`, `data/fixtures/`
- Tests to add first: pilot ZIP priority scoring, pantry eligibility filtering, language and ID constraint handling
- Definition of done: deterministic challenge data snapshot loads locally and ranking logic passes automated tests

### Task 2

- Files to modify: `app/operator/`, `app/resident/`, `lib/alerts.ts`, `lib/analytics.ts`, `components/`
- Tests to add first: resident search happy path, no-match flow, operator publish authorization and stale-data banner
- Definition of done: both core user flows are usable on mobile and backed by passing tests

### Task 3

- Files to modify: `scripts/demo-seed.ts`, `docs/demo-script.md`, `docs/submission-notes.md`, analytics or reporting views
- Tests to add first: smoke test for seed data plus one E2E check that published alerts appear in the resident flow
- Definition of done: the team can run a clean judge demo with sample metrics and a concise story for why the pilot should renew

## Risk Notes

- Partner data freshness is the biggest product-trust risk.
- Resident trust and awareness are the biggest adoption risks.
- If the team tries to solve logistics and delivery in v1, the pilot will likely slip.
