# Claude Notes: StreetCart KC Oracle Focus

Date: 2026-03-28
Repository: ai-prompt-kc-2026

Purpose: capture external feedback and execution guidance without changing the deck file

## Overall Evaluation

Claude's core read was that StreetCart KC has a strong concept because it is:

- unmistakably Kansas City
- built on real infrastructure through the streetcar
- rooted in real local culture through sports fandom
- aimed at a real, legible problem: food access along an existing transit spine

The March Madness tournament mechanic was called out as a particularly strong idea because it gives the concept a repeat-participation loop for volunteer recruitment, which is usually the hardest part of a food distribution program.

## Main Weakness To Fix

The concept is still too broad. The biggest current risk is not idea quality. It is splitting limited time across multiple tracks and submitting several half-built artifacts instead of one polished, judge-readable package.

## Clear Recommendation

Pick Oracle and go deep.

Reasoning:

- Oracle rewards data-backed strategy, budget math, ROI logic, and operational clarity.
- StreetCart KC already has a strong Oracle shape: a `$500K` contract story, a 60-day pilot window, target ZIP codes, and a measurable operating model.
- Muse should support the Oracle story inside the deck instead of becoming a separate full track.

Recommended framing:

> Build one polished Oracle submission. Use selected Muse artifacts only as supporting proof inside that submission.

## What Needs To Exist In The Oracle Submission

- 2-3 concrete data points from the API to prove the problem is real, not anecdotal
- a locked 60-day pilot plan
- budget math that clearly explains where the `$500K` goes
- ROI logic tied to service outcomes and renewal potential
- one intentional slide reserved for curveball response or adaptability
- one slide that explicitly shows how AI was used for ideation, analysis, and asset generation

## Time-Pressure Guidance

Claude's view was that the team should work in three blocks:

1. Lock Oracle first with thesis, pilot, budget, KPIs, and objections handled.
2. Add only the strongest Muse-style supporting assets inside the Oracle package.
3. Keep Architect thin or defer it unless it directly strengthens the same narrative.

## Rubric Read Before Build

### Problem Understanding

Strong potential, but it needs concrete API-backed data points.

### Solution Quality

Strong potential, but it needs a real 60-day operating plan and budget, not just a concept.

### Presentation And Polish

This is likely the deciding factor. A clean deck with one thesis beats a broader, messier submission.

### Adaptability

Plan for this in advance by reserving a pivot slide rather than scrambling later.

### AI Mastery

Potentially strong if the team documents specific uses of AI such as:

- ideation support
- API/data analysis
- budget modeling
- brand and visual exploration

## Practical Implication For This Repo

This feedback reinforces the repo's current direction:

- keep StreetCart KC as one coherent concept
- prioritize Oracle as the main deliverable
- use Muse only where it sharpens Oracle
- avoid building a broad product demo until the Oracle story is locked
